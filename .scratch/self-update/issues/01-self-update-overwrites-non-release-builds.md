# Background daemon self-update overwrites locally-built (non-release) binaries

## Parent

- Reported during QA session

## What's wrong

The background daemon's self-update replaces the running `once` binary whenever
its version string does not exactly match the latest published release tag. A
locally-built binary never matches that tag — a plain build reports its version
as `dev`, and a `make build` reports a `git describe` string like
`v1.2.3-5-gabc1234`. Neither equals the release tag (`v1.2.3`), so the daemon
treats the dev build as out of date, downloads the released asset, and renames it
over the binary on disk.

This makes it impossible to test local changes against a running daemon: you
build your new binary, start the daemon, and within one check interval it has
quietly swapped your work for the public release. The first check happens shortly
after boot (self-update is "due" immediately on a fresh install, with no waiting
period), so the overwrite lands fast and looks like your build "didn't take".

## What I expected

A binary that is not an official release build should never auto-update itself.
The daemon should only self-update when it is actually running a released version
that is older than the latest release. A `dev` build, or a `git describe` build
made from a working tree, should be left alone — the only ways to get a newer
binary in that case are a manual `once self-update` or rebuilding, both of which
the developer is in control of.

## Steps to reproduce

1. Build the binary locally (`make build`, or `go build`).
2. Install/run the background daemon with that binary.
3. Wait one check interval.
4. Observe that the on-disk binary has been replaced by the latest published
   release — your locally-built version is gone.

## Workaround

Setting `ONCE_NO_SELF_UPDATE` to any non-empty value in the daemon's environment
disables the self-update check entirely. This works but is easy to forget and
disables a feature that should still function for real release builds — hence
this issue asks for the daemon to recognise a non-release build on its own.

## Additional context

The self-update "already latest" short-circuit only fires on an *exact* version
match, so it protects the newest release but not a development build, which is the
case that most needs protecting. The fix should gate self-update on the binary
being a genuine release (e.g. treat `dev` and any untagged/dirty `git describe`
build as non-updatable) rather than relying on the exact-match check or the
opt-out environment variable.
