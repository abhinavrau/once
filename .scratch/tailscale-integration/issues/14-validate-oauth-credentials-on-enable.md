# Validate Tailscale OAuth credentials before enabling

## Parent

- Reported during QA session
- PRD: `.scratch/tailscale-integration/PRD.md`

## What's wrong

When enabling Tailscale, Once accepts the OAuth client ID and secret and
proceeds straight to booting `once-tsdproxy` without ever checking that the
credentials actually work. If the credentials are wrong, expired, or lack the
required scopes, the enable flow still reports success — but tsdproxy never
registers a node on the tailnet, and nothing appears. The user is left with a
silently broken tailnet and no indication that the credentials were the problem.

## What I expected

When I enable Tailscale with OAuth credentials, Once should verify them against
the Tailscale API first and refuse to proceed if they're invalid, telling me
clearly that the credentials were rejected (e.g. bad client, missing scope).
The credentials should be validated before any container is booted, so a typo
fails fast instead of leaving a half-configured tailnet.

## Steps to reproduce

1. Enable Tailscale (via `once tailscale enable` or the TUI `t` settings form)
   with an OAuth client ID/secret that is wrong, revoked, or missing the scopes
   Once needs.
2. Observe the enable flow reports success and boots `once-tsdproxy`.
3. Check the tailnet — no node ever registers, and no error explains why.

## Additional context

The OAuth credentials are currently passed through to tsdproxy verbatim and are
only exercised asynchronously when tsdproxy tries to register. There is no
pre-flight check against the Tailscale API. Validating up front turns a silent,
delayed failure into an immediate, actionable error at the moment the user
submits their credentials. This mirrors the existing fail-fast precondition that
already guards the background daemon before enabling.

## Agent Brief

**Category:** bug
**Summary:** Validate the Tailscale OAuth client credentials against the Tailscale API before booting `once-tsdproxy`, and fail the enable flow with a clear error if they're rejected.

**Current behavior:**
Enabling Tailscale accepts the OAuth client ID and secret (via the
`TailscaleSettings` struct that carries `ClientID`/`ClientSecret`) and writes
them verbatim into the tsdproxy YAML config. No call is ever made to the
Tailscale API during enable. tsdproxy performs the OAuth handshake
asynchronously after it boots, so wrong/revoked/under-scoped credentials still
let the enable flow report success — the failure only surfaces later as a node
that never registers, with nothing pointing back to the credentials.

**Desired behavior:**
On enable, before any container is created or the rolling label retrofit runs,
Once must verify the OAuth credentials against the Tailscale API and refuse to
proceed if they're invalid. Validation must catch all three failure modes the
reporter listed: bad/revoked client ID or secret, a missing `auth_keys` scope,
and a missing/invalid tag. The agreed validation strategy is a **mint-and-delete
probe**:

1. Exchange `ClientID`/`ClientSecret` for an access token at the OAuth token
   endpoint (`POST https://api.tailscale.com/api/v2/oauth/token`,
   `grant_type=client_credentials`). A failure here = bad/revoked credentials.
2. Using that token, create a tagged auth key (`POST /api/v2/tailnet/-/keys`),
   then immediately delete it (`DELETE /api/v2/tailnet/-/keys/{id}`). A failure
   here = missing `auth_keys` scope or bad tag config.

This mirrors the existing fail-fast precondition pattern — the daemon
precondition check (`Admin.RequireDaemon()`) that already runs first in
`Namespace.EnableTailscale()`. The credential check should sit in the same
pre-flight position so a typo fails fast and leaves no half-configured tailnet.

**Critical correctness requirement — probe must match tsdproxy's real request:**
Once does **not** set any `tags` field in the tsdproxy provider config today —
tsdproxy mints its tagged auth key internally. Tailscale rejects an untagged
key request from a tagged OAuth client (see tailscale/tailscale#11914), so the
probe's `POST /keys` body must replicate the exact shape tsdproxy:2 uses
(tag set, ephemeral, reusable flags). If the probe and tsdproxy diverge,
validation will either pass-then-fail at runtime or false-reject working
credentials. Verify parity against the existing tsdproxy:2 integration harness:
a credential set the probe accepts must be one tsdproxy can actually register
with, and vice versa.

**Scope note:** This is a Tailscale-SaaS-only check. The hidden headscale
control seam (`controlUrl` + `authKey`, set via `ONCE_TAILSCALE_*` env vars)
replaces OAuth entirely and has no OAuth credentials to validate — the probe
must be skipped when that seam is active, exactly as `buildTSDProxyConfig`
already branches on it.

**Key interfaces:**
- A new Tailscale API client (none exists today — the only HTTP client in the
  Tailscale code talks to tsdproxy's own loopback lookup API, not
  `api.tailscale.com`). Keep it small: token exchange + create key + delete key.
- `Namespace.EnableTailscale(ctx, settings)` — must run the validation before
  network setup / tsdproxy boot / app roll, and return a descriptive error
  (rejected credentials, missing scope, bad tag) that propagates to both the
  CLI (`once tailscale enable`) and the TUI settings form.
- The returned error must distinguish the failure modes so the user knows
  whether it was the credentials, the scope, or the tag.

**Acceptance criteria:**
- [ ] Enabling with a wrong/revoked client ID or secret fails the enable flow
      with a clear "credentials rejected" error and boots no containers.
- [ ] Enabling with credentials that lack the `auth_keys` scope (or a bad tag)
      fails with an error naming the scope/tag problem, not a generic failure.
- [ ] Enabling with valid, correctly-scoped credentials succeeds and proceeds
      exactly as today.
- [ ] The probe auth key is deleted after a successful validation (no stale
      keys accumulate on the tailnet across repeated enables).
- [ ] Validation runs before container creation and the app roll — a rejection
      leaves no proxy/admin container and no half-applied labels.
- [ ] The headscale control seam (`controlUrl`+`authKey`) path skips validation.
- [ ] The CLI and the TUI settings form both surface the rejection message.

**Out of scope:**
- Fetching/persisting the tailnet domain suffix — that's issue #15, which builds
  on the authenticated API call introduced here.
- Re-validating credentials on any path other than enable (e.g. periodic
  re-checks, daemon-side validation).
- Headscale credential validation.
