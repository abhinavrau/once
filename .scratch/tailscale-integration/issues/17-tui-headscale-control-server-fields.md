# TUI: offer Headscale control server URL + auth key in the Tailscale settings form

## Parent

- Reported during QA session
- PRD: `.scratch/tailscale-integration/PRD.md`

## What's wrong

The TUI Tailscale settings form (the `t` overlay) only offers OAuth client ID
and secret. A user who wants to connect their tailnet to a self-hosted Headscale
control server has no way to enter the control server URL and auth key from the
TUI — they can only configure OAuth against Tailscale SaaS.

## What I expected

The Tailscale settings form should let me choose between Tailscale SaaS (OAuth)
and a self-hosted Headscale control server, and when I pick Headscale, enter its
URL and an auth key. Saving should enable the tailnet against that control
server, and re-opening the form should pre-populate what I previously entered.

## Steps to reproduce

1. Open the global Tailscale settings form (the `t` overlay).
2. Look for fields to enter a Headscale control server URL and auth key.
3. There are none — only OAuth client ID and secret are shown.

## Additional context

This is the TUI surface for the Headscale control-server option; the settings
model, lifecycle, and CLI surface are covered by #16. The two modes (SaaS OAuth
vs. Headscale control server + auth key) are mutually exclusive, so the form
needs to make clear which one is in effect.

## Blocked by

- #16 — needs the persisted control-server settings model and enable lifecycle.

## Agent Brief

**Category:** enhancement
**Summary:** Add Headscale control-server fields to the global Tailscale settings form (`t` overlay): let the user choose between Tailscale SaaS (OAuth) and a self-hosted control server, enter the control URL + auth key, save, and have the form pre-populate on re-open.

**⚠️ Blocked by #16.** This is the TUI surface for the persisted control-server
settings model and enable lifecycle that #16 introduces. Do not start until #16
has landed — the form depends on the new `TailscaleSettings.ControlURL`/`AuthKey`
fields and the mode-aware enable path.

**Current behavior:**
The global Tailscale settings form offers only OAuth client ID and client
secret. There is no way from the TUI to point the tailnet at a self-hosted
Headscale control server; the control-server path is CLI/settings-only after #16.

**Desired behavior:**
The form lets the user pick which mode is in effect — Tailscale SaaS (OAuth) or a
self-hosted Headscale control server — and shows the relevant fields for the
chosen mode: OAuth client ID + secret, or control server URL + auth key. The two
modes are mutually exclusive, and the form makes clear which one is active.
Saving enables the tailnet in the selected mode (calling the same enable path the
CLI uses, populating the appropriate `TailscaleSettings` fields). Re-opening the
form pre-populates whichever mode and values were previously saved.

**Key interfaces:**
- The Tailscale settings form (currently building OAuth-only fields and calling
  `EnableTailscale` with `ClientID`/`ClientSecret`) — add a mode selector and the
  control URL / auth key inputs, and populate the corresponding
  `TailscaleSettings` fields based on the selected mode.
- It must load existing settings to pre-populate: if a control URL is persisted,
  open in control-server mode with those values; otherwise OAuth mode with the
  existing client ID/secret.
- Submission goes through the same mode-aware enable path #16 establishes — the
  form should not re-implement mode resolution, just supply the right fields.

**Acceptance criteria:**
- [ ] The form offers a clear choice between SaaS OAuth and Headscale
      control-server mode.
- [ ] Selecting Headscale mode reveals control server URL + auth key fields;
      selecting OAuth mode reveals client ID + secret fields.
- [ ] Saving in Headscale mode enables the tailnet against that control server
      (control URL + auth key persisted, no OAuth values sent).
- [ ] Saving in OAuth mode behaves exactly as the form does today.
- [ ] Re-opening the form pre-populates the previously saved mode and values.
- [ ] The form does not allow submitting both modes at once.

**Out of scope:**
- The settings model, persistence, precedence, and CLI surface — all #16.
- Credential/connectivity validation of either mode — #14.
- Funnel and per-app settings (unrelated forms).
