# Tailscale status says "no nodes" without surfacing why registration failed

## Parent

- Reported during QA session
- PRD: `.scratch/tailscale-integration/PRD.md`

## What's wrong

When the tailnet has no registered nodes, `once tailscale status` prints a flat
"No tailnet nodes registered" and stops. That message is identical whether the
tailnet is still spinning up, has no apps deployed, or is permanently broken
because every registration is being rejected (e.g. a tag the control plane won't
accept). The information that would actually explain the empty list — the helper
container's registration error — exists and is logged by the helper, but `once`
never surfaces it, so the user is left guessing and re-running with no new signal.

## What I expected

When there are zero nodes, `once tailscale status` should help me distinguish
"not ready yet" from "misconfigured and will never work". If the helper is
reporting a registration failure (a rejected tag, a bad/expired credential, a
control-plane error), status should surface that reason — and point me at where
to fix it — instead of a bare "no nodes" line that looks the same in every case.

## Steps to reproduce

1. Enable Tailscale with a tag the tailnet will not accept for node registration
   (see the companion enable-validation issue).
2. Deploy an app.
3. Run `once tailscale status`. Observe only "No tailnet nodes registered", with
   no indication that registrations are actively being rejected — even though the
   helper container's logs show the exact rejection.

## Additional context

This is a diagnosability issue independent of the enable-time check: even after
enable is made stricter, a tailnet can degrade later (a tag removed from the ACL,
a credential revoked) and leave the same silent empty list. Status is the natural
place to expose the underlying registration error so the user can act without
having to inspect helper-container logs by hand.
