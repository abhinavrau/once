# Tailscale enable passes its credential check but no nodes can ever register

## Parent

- Reported during QA session
- PRD: `.scratch/tailscale-integration/PRD.md`

## What's wrong

`once tailscale enable` completes successfully — tsdproxy boots, apps are rolled,
the command returns no error — yet not a single tailnet node ever registers. After
enabling and deploying an app (e.g. a whoami container), `once tailscale status`
reports "No tailnet nodes registered" indefinitely.

The cause is that the enable-time credential check proves the wrong thing. It
mints a tagged auth key and deletes it, treating a successful mint as proof the
tag works. But the tailnet enforces tag ownership (the ACL `tagOwners` policy)
only when a node actually *registers* with the key — not when the key is minted.
So a tag the OAuth client can mint keys for, but that is not listed in
`tagOwners`, sails through the pre-flight and then fails every real registration
with "requested tags are invalid or not permitted". Enable reports success on a
configuration that can never produce a working node.

This was confirmed live: minting the probe key for the configured tag returns
success, while tsdproxy's real node registration with the same OAuth client and
the same tag is rejected by the control plane.

## What I expected

If the configured tag cannot be used to register a node on the tailnet, enable
should fail (or at least clearly warn) at enable time, naming the tag and the tag
ownership requirement — not report success and leave me with a permanently empty
tailnet. The check that gates enable should reflect whether a node can actually
come up, since minting a key is not sufficient evidence of that.

## Steps to reproduce

1. Use an OAuth client that can mint keys for a tag (e.g. `tag:once`) but where
   that tag is NOT listed in the tailnet ACL `tagOwners`.
2. Run `once tailscale enable` with that tag. Observe it succeeds.
3. Deploy an app (e.g. a whoami container).
4. Run `once tailscale status`. Observe "No tailnet nodes registered", and that it
   never changes — while the tsdproxy helper's own logs show the registration
   being rejected for the tag.

## Additional context

The pre-flight probe was designed to stay in lockstep with the helper's real
auth-key request so that "passes the probe" implies "works in production". This
gap breaks that promise: tag-ownership rejection happens at node registration, a
step the mint-and-delete probe never reaches. Closing it likely means verifying
registration itself (e.g. confirming at least one node comes up shortly after
enable, or otherwise checking the tag against the ownership policy) rather than
inferring success from a key mint.
