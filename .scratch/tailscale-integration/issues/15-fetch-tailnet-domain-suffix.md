# Fetch the tailnet domain suffix from the Tailscale API on enable

## Parent

- Reported during QA session
- PRD: `.scratch/tailscale-integration/PRD.md`

## What's wrong

Once never learns the tailnet's own domain suffix (the MagicDNS base, e.g.
`tailnet-name.ts.net`) directly from Tailscale. Today the only way it sees a
tailnet URL is second-hand, by reading back what tsdproxy reports after a node
has already registered. Until something registers, Once has no idea what the
tailnet is called, so it can't show the user where their apps will live or
construct tailnet URLs ahead of time.

## What I expected

Right after the OAuth credentials are validated on enable, Once should ask the
Tailscale API for the tailnet's domain suffix and remember it. With the suffix
known up front, Once can show the user the tailnet they're connected to and
build each app's `*.ts.net` Magic DNS URL without waiting on tsdproxy's lookup.

## Steps to reproduce

1. Enable Tailscale with valid OAuth credentials.
2. Before any app has registered a node, look for the tailnet's domain suffix
   (the `*.ts.net` base) anywhere in Once.
3. There is none — Once only surfaces a tailnet URL once tsdproxy has already
   registered and reported it back.

## Additional context

The domain suffix is a property of the tailnet itself and is available from the
Tailscale API as soon as the OAuth credentials authenticate, independent of
whether any node has registered. Fetching and persisting it on enable removes
the dependency on tsdproxy's lookup for knowing the tailnet's name, and gives a
stable basis for displaying and constructing tailnet URLs. Where the suffix is
stored and surfaced (alongside the other Tailscale settings, in the TUI details,
etc.) is open for the implementer.

## Agent Brief

**Category:** enhancement
**Summary:** On enable, learn the tailnet's MagicDNS domain suffix (e.g. `tailnet-name.ts.net`) directly from Tailscale and persist it, so Once can show the tailnet and construct app `*.ts.net` URLs without waiting for tsdproxy to register a node.

**⚠️ Blocked by #14.** This builds directly on the validated, authenticated
Tailscale API path introduced in #14. Do not start until #14 has landed.

**Current behavior:**
Once never learns the tailnet's own domain suffix from Tailscale. The only way
it sees a tailnet URL today is second-hand: after a node registers, it reads
the URLs back from tsdproxy's loopback lookup API. Until something registers,
Once cannot show the user what tailnet they're on or build any `*.ts.net` URL.

**Desired behavior:**
Right after #14 validates the OAuth credentials on enable, Once should learn the
tailnet's domain suffix and persist it, so the suffix is known immediately —
before any app node registers. The agreed mechanism is to **extend #14's
mint-and-delete probe**: with the auth key #14 already mints, bring up a
short-lived **ephemeral** Tailscale node, read its MagicDNS name (of the form
`<host>.<suffix>`, e.g. `probe-xxxx.tailnet-name.ts.net`), derive the suffix by
stripping the leading host label, then tear the node down. Ephemeral nodes
self-deregister, so this leaves nothing behind on the tailnet.

Folding this into #14's existing probe is preferred: a single probe per enable
that both validates registration end-to-end **and** yields the suffix. The
`tsnet` userspace approach (bring up a node with the key, read `Self.DNSName`
from its status, close it) is the natural in-process mechanism, but the exact
node implementation is the implementer's call.

**Persistence and surfacing:**
- Add a `DomainSuffix` field to the global `TailscaleSettings` struct (the one
  already carrying `ClientID`/`ClientSecret`/`APIKey`, persisted as the `once`
  JSON label on the `once-tsdproxy` container). It serializes for free via its
  JSON tag.
- Once the suffix is stored, the places that currently build/show tailnet URLs
  from tsdproxy's lookup (the dashboard's Tailscale info polling and the app
  details "Tailnet URL" row) can construct `https://<app>.<suffix>` from the
  stored suffix instead of depending solely on tsdproxy having registered.
  Constructing URLs from the stored suffix is in scope; reworking every consumer
  is not — at minimum persist it and make it available.

**Scope note (headscale seam):** As in #14, the hidden `controlUrl`+`authKey`
control seam is not Tailscale SaaS. If the probe is skipped there (per #14),
the suffix fetch is skipped too; do not attempt a SaaS API call on that path.

**Key interfaces:**
- `TailscaleSettings` — new `DomainSuffix string` field (`json:"domainSuffix,omitempty"`).
- The Tailscale API client / probe from #14 — extended to register an ephemeral
  node and return its derived suffix alongside the validation result.
- `Namespace.EnableTailscale(ctx, settings)` — persists the fetched suffix into
  the settings it stores on the tsdproxy container.

**Acceptance criteria:**
- [ ] After a successful enable with valid credentials, the tailnet domain
      suffix is persisted in `TailscaleSettings.DomainSuffix`.
- [ ] The suffix is known without any app node having registered first.
- [ ] The ephemeral probe node and its key are gone after enable (no stale
      device or key on the tailnet).
- [ ] Tailnet URLs for apps can be constructed as `https://<app>.<suffix>` from
      the stored suffix.
- [ ] The headscale control seam path does not attempt the SaaS suffix fetch.
- [ ] A failed suffix fetch is surfaced/handled sensibly and does not silently
      leave a half-configured tailnet (consistent with #14's fail-fast posture).

**Out of scope:**
- The credential validation itself (#14).
- Reworking every URL consumer in the TUI/CLI — persisting the suffix and making
  URL construction possible is enough; broader display polish is separate.
- Headscale/`controlUrl` suffix discovery.
