# Let users enable Tailscale against a self-hosted Headscale control server

## Parent

- Reported during QA session
- PRD: `.scratch/tailscale-integration/PRD.md`

## What's wrong

Enabling Tailscale only works against Tailscale's hosted service via OAuth
credentials. Users who run their own Headscale control server have no supported
way to point Once at it. The capability technically exists — once-tsdproxy can
be configured with a control server URL and auth key instead of OAuth — but it's
only reachable through undocumented environment variables used by the test
harness, and the chosen control server is not persisted with the rest of the
Tailscale settings.

## What I expected

When enabling Tailscale, I should be able to opt to use my own Headscale control
server by providing its URL and an auth key, as an alternative to Tailscale SaaS
OAuth credentials. Once should use those instead of OAuth, and remember them so
the tailnet still works across restarts without re-supplying them through the
environment.

## Steps to reproduce

1. Run `once tailscale enable` (or open the TUI settings form).
2. Look for a way to provide a Headscale control server URL and auth key instead
   of Tailscale OAuth credentials.
3. There is none — only OAuth client ID/secret are offered, and the control-URL
   path is reachable only via undocumented environment variables.

## Additional context

This is the first-class promotion of the existing hidden control-server seam:
when a control URL + auth key are supplied they replace OAuth entirely (OAuth is
Tailscale-SaaS-only and can't reach a Headscale control plane). The two modes are
mutually exclusive. This issue covers the durable settings model (persisting the
control URL + auth key alongside the OAuth fields) and the CLI enable surface;
the TUI surface is a separate issue. Validation/credential-checking of the chosen
mode is tracked separately (#14).

## Agent Brief

**Category:** enhancement
**Summary:** Promote the hidden control-server seam to a first-class, persisted option: let `once tailscale enable` target a self-hosted Headscale control server via a control URL + auth key instead of Tailscale SaaS OAuth, and remember the choice across restarts.

**Current behavior:**
tsdproxy can already be pointed at a control server: when a control URL and auth
key are supplied, the rendered tsdproxy config uses `controlUrl` + `authKey`
instead of the OAuth `clientId`/`clientSecret` block (the two are mutually
exclusive — OAuth is Tailscale-SaaS-only and cannot reach a Headscale control
plane). But today those values are only read from the undocumented environment
variables `ONCE_TAILSCALE_CONTROL_URL` / `ONCE_TAILSCALE_AUTH_KEY` at enable
time, used by the headscale test harness. They are not persisted with the rest
of the Tailscale settings, and the CLI hard-requires `--client-id` /
`--client-secret`, so there is no supported, durable way to use Headscale.

**Desired behavior:**
A user can run `once tailscale enable` against a Headscale control server by
supplying its URL and an auth key as an alternative to OAuth. The chosen control
server is persisted alongside the other Tailscale settings, so the tailnet keeps
working across container restarts and re-enables without re-supplying anything
through the environment. OAuth mode and control-server mode remain mutually
exclusive.

**Settings precedence (decided):** The environment variables remain the highest
precedence so the existing headscale harness keeps working unchanged. Resolution
order when building the tsdproxy config is:
`ONCE_TAILSCALE_* env vars > persisted ControlURL/AuthKey > OAuth clientId/secret`.
Persisted settings are the durable source of truth for normal use; the env vars
are a harness/override layer that wins when present.

**Key interfaces:**
- `TailscaleSettings` — add `ControlURL string` and `AuthKey string` fields
  (JSON-tagged, e.g. `controlUrl` / `authKey`), persisted in the same `once`
  label on the `once-tsdproxy` container as the OAuth fields. `omitempty` so
  OAuth-mode settings are unchanged on disk.
- The tsdproxy config builder (currently `buildTSDProxyConfig`, which today takes
  the env-var control URL/auth key as separate args) — should resolve the
  control URL/auth key by the precedence above (env, then persisted settings),
  and emit the `controlUrl`/`authKey` provider block when a control URL is in
  effect, otherwise the OAuth block. It must not emit both.
- `Tailscale.Enable()` recreate check — it currently decides whether to recreate
  the container by comparing only `ClientID`/`ClientSecret`. Extend it so a
  change to `ControlURL`/`AuthKey` (or a switch between modes) also triggers
  recreation; an unchanged control-server config should still short-circuit to
  `ensureRunning`.
- CLI `once tailscale enable` — add `--control-url` and `--auth-key` flags.
  `--client-id`/`--client-secret` must no longer be unconditionally required;
  instead enforce that exactly one mode is provided (OAuth pair *or*
  control-url+auth-key), e.g. via cobra's mutually-exclusive / required-together
  flag groups. The command should map the chosen flags into the new
  `TailscaleSettings` fields.

**Acceptance criteria:**
- [ ] `once tailscale enable --control-url <url> --auth-key <key>` enables the
      tailnet against that control server, with no OAuth credentials supplied.
- [ ] The control URL and auth key are persisted in the tsdproxy container's
      `once` label and survive a container restart / process restart without
      re-supplying the environment variables.
- [ ] `once tailscale enable --client-id <id> --client-secret <secret>` still
      works exactly as before (OAuth mode).
- [ ] Supplying both modes' flags together is rejected with a clear error;
      supplying neither is rejected.
- [ ] Changing the persisted control URL or auth key on a subsequent enable
      recreates the tsdproxy container; an unchanged config does not.
- [ ] With `ONCE_TAILSCALE_CONTROL_URL`/`ONCE_TAILSCALE_AUTH_KEY` set, behavior
      is unchanged from today (env wins over persisted settings) — the headscale
      harness still passes.
- [ ] The rendered tsdproxy config never contains both the OAuth block and the
      controlUrl/authKey block.

**Out of scope:**
- The TUI surface for choosing/entering Headscale settings — that's #17.
- Validation/credential-checking of either mode — tracked by #14 (and #14's
  brief already skips the SaaS probe when the control seam is active).
- Any change to how the OAuth path itself works.
