# Normalize or validate the Tailscale tag before probing

## Parent

- Reported during QA session
- PRD: `.scratch/tailscale-integration/PRD.md`

## What's wrong

When enabling Tailscale, Once passes the tag the user provides straight to the
Tailscale API without checking or normalizing its format. Tailscale only accepts
tags in `tag:name` form, so a user who supplies the bare tag name (e.g. `once`
instead of `tag:once`) gets the credential probe rejected with a raw, confusing
Tailscale error:

```
Error: Tailscale rejected the probe auth key for tag "once" (400 Bad Request):
requested tags [once] are invalid or not permitted; ensure your OAuth client has
the auth_keys scope and owns this tag (Access Controls → tagOwners)
```

The user did everything else right — valid OAuth client, a tag the client owns
in `tagOwners` — but because the tag was entered without the `tag:` prefix, the
probe fails and the guidance sends them chasing a scope/ownership problem that
doesn't exist. The `requested tags [once]` in the response (no `tag:` prefix)
shows the bare name reaching the API verbatim.

## What I expected

When I enter a tag without the `tag:` prefix, Once should either normalize it to
`tag:once` for me, or reject it up front with a clear message that tags must be
in `tag:name` form — before it ever probes Tailscale. The confusing
scope/ownership error should only appear when the scope or ownership is actually
the problem, not when the tag is simply missing its prefix.

## Steps to reproduce

1. Enable Tailscale (via `once tailscale enable --tag once` or the TUI `t`
   settings form) with a valid OAuth client that owns `tag:once`, but enter the
   tag as `once` rather than `tag:once`.
2. Observe the enable flow fails with "requested tags [once] are invalid or not
   permitted", pointing at the OAuth scope and tagOwners.
3. Re-run with `tag:once` and observe it works — confirming the only difference
   was the missing prefix.

## Additional context

The TUI field is labelled "Tag the OAuth client owns, e.g. tag:once", so the
`tag:` prefix is implied but not enforced. The tag is threaded verbatim through
to both the credential probe and the tsdproxy config, so a bare name fails the
same way in both places. Normalizing (prepend `tag:` when absent) or validating
the format before the probe turns a misleading API rejection into either a
transparent success or an immediate, accurate error. This sits alongside the
fail-fast OAuth credential probe added in issue #14.

## Agent Brief

**Category:** bug

**Summary:** Normalize the user-supplied Tailscale tag to `tag:name` form before
it is used, so a bare tag name (e.g. `once`) no longer produces a misleading
Tailscale scope/ownership rejection.

**Current behavior:**
`settings.Tag` is threaded verbatim from the CLI `--tag` flag and the TUI
settings form into two consumers: the enable-time credential probe (which splits
it on commas and sends it to `POST /tailnet/-/keys`) and the tsdproxy provider
config. The only validation is a non-empty check. A tag entered without the
`tag:` prefix reaches the Tailscale API as a bare name, which Tailscale rejects
with `requested tags [once] are invalid or not permitted`. Once wraps that in
guidance about the `auth_keys` scope and `tagOwners`, sending the user to chase a
permission problem that doesn't exist.

**Desired behavior:**
Before the tag is used by either consumer, normalize it: split on commas, trim
surrounding whitespace from each segment, and prepend `tag:` to any segment that
does not already start with `tag:`. Already-prefixed segments pass through
unchanged, so mixed input like `once,tag:admin` becomes `tag:once,tag:admin`. The
normalized value must feed **both** the credential probe and the tsdproxy config,
so the two never diverge. After normalization the existing probe still validates
real tag ownership — normalization only fixes the prefix, it does not bypass the
scope/ownership check.

**Key interfaces:**
- `TailscaleSettings.Tag` — the field carrying the tag from CLI/TUI through to
  the probe and tsdproxy config.
- `Namespace.EnableTailscale(ctx, settings)` — normalize once here (or via a
  `TailscaleSettings` accessor/normalize step it calls) so every downstream
  reader sees the prefixed value. Normalizing at a single choke point is the
  root-cause fix; do not patch only the probe path and leave the config path
  un-normalized.
- The empty-tag error in `validateOAuthCredentials` stays as-is — an empty tag is
  still an error, normalization applies only to non-empty input.

**Acceptance criteria:**
- [ ] Enabling with `--tag once` succeeds (assuming valid, owned `tag:once`),
      with the probe and tsdproxy config both seeing `tag:once`.
- [ ] Enabling with `--tag tag:once` continues to work unchanged.
- [ ] Mixed input `once,tag:admin` normalizes per-segment to
      `tag:once,tag:admin`.
- [ ] An empty tag still fails with the existing "a Tailscale tag is required"
      error.
- [ ] The genuine scope/ownership rejection (a properly-prefixed tag the client
      doesn't own) still surfaces its existing error — normalization does not
      mask it.
- [ ] Unit-tested with an httptest server (SaaS probe paths can't run in CI per
      AGENTS.md): assert the normalized tag is what reaches the create-key
      request body.

**Out of scope:**
- Validating that the tag actually exists in the tailnet ACL beyond what the
  existing probe already checks.
- Any change to the headscale control seam, which skips the OAuth probe.
- Re-formatting or normalizing tags on any path other than enable.
