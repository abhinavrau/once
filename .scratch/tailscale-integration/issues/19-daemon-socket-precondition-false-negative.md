# Tailscale enable reports the daemon down when it is actually running

## Parent

- Reported during QA session
- PRD: `.scratch/tailscale-integration/PRD.md`

## What's wrong

Running `once tailscale enable` fails its daemon precondition check with:

```
Error: once-admin socket /var/run/once-admin.sock not found; is the background
daemon running?
```

even though the background daemon **is** running. The message blames the daemon
being down and points the user at starting it, but the daemon is already up — so
the user is sent to fix a problem that doesn't exist, with no hint about what the
check actually disagrees on.

The socket path the check looks for is derived (it honours
`ONCE_ADMIN_RUNTIME_DIR`, defaults to `/var/run`) and the check also requires a
companion `once-admin-nginx.conf` next to the socket. So the daemon can be
genuinely running while the precondition looks at a different path — or finds the
socket but not its config file — and reports the same generic "is the daemon
running?" error in every case. The message always names `/var/run/once-admin.sock`
regardless of the path actually checked, which hides the real mismatch.

## What I expected

When the background daemon is running, `once tailscale enable` should pass the
precondition. If the precondition genuinely can't reach the admin socket, the
error should tell me *why* — the exact path it looked for and whether the socket
or its config file was missing — instead of flatly asserting the daemon is down
when it isn't. The enable flow and the running daemon should agree on where the
socket lives (same namespace, same runtime dir) so a running daemon is never
reported as missing.

## Steps to reproduce

1. Install and start the background daemon (`once background install`) and
   confirm it is running.
2. Run `once tailscale enable` with valid settings.
3. Observe the enable flow fails with "once-admin socket
   /var/run/once-admin.sock not found; is the background daemon running?" even
   though the daemon is up.

## Additional context

The same daemon serves the admin socket and the funnel auto-expiry loop, and the
once-admin container bind-mounts the socket + nginx config the daemon writes. The
precondition guards against a broken mount by requiring both files before
booting the container. Likely triggers for the false negative: the enable command
and the daemon resolving different socket paths (a `ONCE_ADMIN_RUNTIME_DIR`
difference, or a different `--namespace`), or the socket present but the
`once-admin-nginx.conf` companion missing. Whichever it is, the error should name
the path and the missing piece rather than blaming the daemon being down.

## Agent Brief

**Category:** bug

**Summary:** The daemon's admin socket server is an unsupervised goroutine whose
failure leaves the daemon "active" with no socket, so `once tailscale enable`'s
precondition fails with a misleading "is the daemon running?" error while the
daemon is in fact running. Supervise the socket server in-process, and make the
precondition error report what it actually found.

**Current behavior:**
The background runner starts the admin socket server with
`go func() { if err := NewAdminServer().Run(ctx); err != nil { slog.Error(...) } }()`.
On any error from `Run` (a stale socket left by an unclean shutdown, a listen
race at boot, a transient FS error writing the nginx config) the goroutine logs
and exits, but the runner's main loop keeps running. systemd sees the process
alive and reports the service `active`, yet no socket is published. The
enable-time precondition (`Admin.RequireDaemon`) then returns "once-admin socket
<path> not found; is the background daemon running?" — pointing the user at a
daemon that is genuinely running. `RequireDaemon` checks an
`ONCE_ADMIN_RUNTIME_DIR`-derived path and also requires the nginx config file
beside the socket, but the message blames the daemon being down in every case.

**Desired behavior:**
1. **Supervise the socket server (root cause).** The admin socket server must not
   be able to die silently while the daemon stays up. Run it under an in-process
   supervisor: if `AdminServer.Run` returns while the context is still live, log
   the error and restart it with bounded backoff, until the context is cancelled.
   A transient failure at boot must self-heal so the socket reliably appears
   without a manual daemon bounce. (Maintainer chose in-process retry over
   exiting the process for systemd to restart.)
2. **Honest precondition error.** `RequireDaemon`'s message should report what it
   actually checked: the exact path it looked for and whether the **socket** or
   the **nginx config** was missing, and stop asserting the daemon is down as the
   sole cause (it may be up with a failed socket server — point the user at the
   daemon logs, e.g. the "Admin socket server" error line). Keep the
   socket-vs-config distinction that already exists.

**Key interfaces:**
- `Runner.Run` (`internal/background/runner.go`) — replace the fire-and-forget
  `go NewAdminServer().Run(ctx)` with a supervised restart loop honouring
  `ctx` cancellation for clean shutdown.
- `AdminServer.Run` (`internal/background/admin.go`) — already returns its error
  and removes the socket on ctx cancel; the supervisor consumes that return.
- `Admin.RequireDaemon` (`internal/docker/admin.go`) — reword the socket/config
  "not found" errors to name the path and not pin the cause solely on the daemon
  being stopped.

**Acceptance criteria:**
- [ ] If the admin socket server fails its first listen (e.g. a pre-existing
      stale socket or transient error), the daemon retries and the socket
      eventually appears without restarting the service — unit-testable by
      forcing `Run` to fail once then succeed and asserting the socket is
      published.
- [ ] The supervisor stops cleanly on context cancellation and does not leak a
      goroutine or busy-loop (bounded backoff, not a tight retry).
- [ ] `RequireDaemon`'s error names the path it checked and distinguishes a
      missing socket from a missing nginx config, without asserting the daemon is
      stopped as the only explanation.
- [ ] A genuinely-stopped daemon (no socket, no config) still fails the
      precondition with an actionable message.

**Out of scope:**
- Changing the socket path derivation or the `ONCE_ADMIN_RUNTIME_DIR` contract.
- Adding namespace isolation or hardening to the systemd unit.
- The tag-normalization work (issue #18).
